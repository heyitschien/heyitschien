# Model and Tool Attribution

## Principle

My projects are AI-assisted and human-directed.

I believe attribution should make two truths visible at the same time:

1. advanced AI systems materially contribute reasoning, alternatives, implementation, review, and acceleration;
2. I remain responsible for problem framing, synthesis, decisions, verification, and the final accepted result.

I do not present model-generated work as if no AI assistance occurred. I also do not treat persuasive model output as verified merely because it sounds complete.

## Typical role map

The exact role varies by project and model version. This map describes the recurring pattern rather than claiming that every tool performed every function on every project.

### ChatGPT

Common contributions:

- research synthesis;
- systems reasoning;
- architecture translation;
- design critique;
- gap analysis;
- documentation review;
- communication refinement;
- decision clarification.

### Gemini

Common contributions:

- independent cross-validation;
- alternate perspectives;
- source exploration;
- challenging conclusions produced by another model;
- comparing possible approaches.

### Cursor and coding agents

Common contributions:

- implementation;
- refactoring;
- repository changes;
- tests;
- documentation updates;
- repetitive engineering execution;
- reconciling scoped work against repository context.

### GitHub

System role:

- versioned engineering memory;
- source control and provenance;
- issues and pull requests;
- review history;
- CI and validation records;

- public evidence for selected work.

### GitHub Issues and Linear

System role:

### Chien Escalera Duong

## Attribution by evidence

Where practical, project documentation should identify material tool and model contributions. Attribution should be based on actual evidence such as:

- issue and pull-request history;
- project documentation;
- implementation logs;
- decision records;
- test results;
- commit history;
- human review notes.

A model name should not be added merely for prestige. A tool should be credited for the role it actually performed.

## Human judgment and validation

AI assistance does not remove the need for:

- source verification;
- security and privacy judgment;
- user-centered decisions;
- testing;
- domain review;
- legal or professional review when appropriate;
- accountability for errors.

A design may be approved for implementation while still awaiting empirical validation. Evidence can overturn my decision or any model recommendation.

## Public portfolio boundaries

Public documentation should not expose:

- credentials or secrets;
- private customer or employment data;
- proprietary trading logic;
- collaborator-sensitive information;
- private repository contents;
- model conversations that contain confidential material.

Public case studies should show the operating method and verifiable outcomes without disclosing protected inputs.

## Why this matters

Transparent attribution is not a disclaimer that weakens the work. It explains the capability accurately.

The value I provide is not pretending to be the model. It is learning how to direct multiple forms of machine intelligence, documentation, software tools, and human judgment toward a coherent and testable outcome.
