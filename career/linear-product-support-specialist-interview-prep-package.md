## 8. Likely interview questions and strong answer patterns

### Q1. Tell me about yourself.

Use the 2-minute answer.

### Q2. Why Linear?

> Linear is one of the clearest examples of software that respects how builders actually work. It reduces coordination drag without becoming heavy. I like that GitHub, Slack, Figma, and preview deployments can remain specialized tools, while Linear becomes the shared source of truth for status, ownership, and decisions.

### Q3. Why Product Support instead of engineering?

> I am interested in the bridge. I enjoy building, but I also enjoy helping people understand systems. Product Support lets me stay close to users, product behavior, and technical workflows. I see it as a role where clear writing, empathy, troubleshooting, and product judgment all matter.

### Q4. What is your biggest gap for this role?

> My background is not the traditional SaaS support path, so I have to be intentional about proving the workflows. That is why I created a Linear practice workspace and support lab. I am also training myself to be more concise: define the issue, gather evidence, explain the likely cause, and propose the next step.

### Q5. How would you handle an angry or confused customer?

> First I would acknowledge the impact without overpromising. Then I would clarify what they expected, what actually happened, and what environment or workflow they were using. I would try to reproduce or narrow the issue. If I can solve it, I would give clear steps. If I need engineering, I would escalate with a concise summary, reproduction steps, screenshots/logs if available, and customer impact.

### Q6. A GitHub PR did not link to a Linear issue. What do you check?

> I would check whether the GitHub integration is enabled, whether the repo is connected to the Linear workspace, whether the branch name or PR title/description includes the correct Linear issue ID, whether they used a linking keyword versus a closing keyword, and whether the PR state or permissions affect the sync. I would also check whether the user is looking at the right team, issue, or view.

### Q7. What is the difference between GitHub Issues and Linear?

> GitHub Issues are close to code and open-source workflows. Linear is a broader work coordination system. GitHub is where code and PR history live; Linear is where the team tracks ownership, priority, status, projects, milestones, blockers, customer context, and cross-functional visibility.

### Q8. How do you decide whether something is a bug or expected behavior?

> I compare the user’s expectation with documented behavior and product design. Then I look for reproduction steps. If the system behaves as designed but the expectation is reasonable, I would explain the current behavior clearly and capture feedback. If it behaves inconsistently or breaks a promised workflow, I would treat it as a bug and escalate with evidence.

### Q9. How do you communicate technical issues to non-technical users?

> I start with the outcome they care about. Then I explain the cause in plain language, avoid internal jargon, and give the next action. If technical detail is useful, I separate it into a short “what happened” section rather than leading with it.

### Q10. What does ownership mean to you?

> Ownership means I do not just answer the first message and disappear. I understand the problem, clarify ambiguity, move it toward resolution, communicate progress, and leave the system better if I notice a repeat pattern.

### Q11. Tell me about a time you learned something quickly.

Use software career pivot or Linear workspace setup.

### Q12. Tell me about a time you handled pressure.

Use stunt safety or rideshare customer situation.

### Q13. How do you manage ambiguity?

> I write down what is known, what is unknown, what the user expected, and what evidence would change my mind. Then I choose the smallest next step that reduces uncertainty.

### Q14. How do you prioritize support issues?

> I would consider customer impact, severity, number of affected users, whether there is a workaround, whether it blocks core workflow, and whether it has strategic/account importance. I would also separate urgent from important: some issues are emotionally loud but low severity, while others are quiet but blocking many users.

### Q15. How would you contribute to product feedback?

> I would look for repeated support patterns: the same confusion, same setup mistake, same missing affordance, or same documentation gap. Then I would summarize the pattern with examples, not just opinions.

## 9. Linear product mental model

### Simple model

Linear is the shared source of truth for work.

- **Issue:** One unit of work.
- **Project:** A larger outcome composed of issues.
- **Milestone:** A phase or checkpoint inside a project.
- **Triage:** Intake queue before something becomes active work.
- **Labels:** Extra context like customer-facing, engineering-escalation, GitHub, docs.
- **Priority:** Importance/urgency signal.
- **Cycles:** Team planning rhythm.
- **Initiatives:** Higher-level focus area across projects.
- **Docs:** Long-form project or process context.
- **Inbox:** Notification and attention center.

### How to explain the GitHub integration

> GitHub is where the code changes happen. Linear listens to GitHub activity and reflects the status back on the issue. If the branch name, PR title, or PR description references the Linear issue ID, Linear can link the PR. If a closing keyword is used and configured, merging the PR can close the issue.

### How to explain the Slack integration

> Slack is where conversations often start. Linear lets teams turn those conversations into trackable issues, then sync comments and status back so the loop closes without everyone constantly switching tools.

### How to explain cross-functional value

> Engineers can work in GitHub. Designers can work in Figma. Support can work from customer context. Product can track status. Linear pulls the thread together so one issue tells the story.

## 10. Support case playbooks

### Case A — “My GitHub PR did not sync to Linear.”

**Clarify:**

- Which Linear workspace and team?
- Which issue ID?
- Which GitHub repo and PR?
- Did they expect link only or auto-close?

**Check:**

1. GitHub integration enabled for the workspace/org.
2. Repo connected and permissions granted.
3. Branch name includes issue ID, e.g. `HEY-13-fix-sync`.
4. PR title/body includes correct issue ID.
5. Keyword used: `Part of HEY-13` vs `Closes HEY-13`.
6. Closing keywords enabled in integration settings.
7. User is viewing the correct team/project/view.
8. PR state: draft/open/merged.

**Customer reply pattern:**

> Thanks — I can help narrow this down. The most common causes are the PR not referencing the Linear issue ID, the repo not being connected to the workspace, or the PR using a non-closing keyword when you expected the issue to close. Can you send the Linear issue ID and the GitHub PR link? I’ll check whether the integration has enough information to connect them.

**Engineering escalation note:**

```md
Customer reports GitHub PR not syncing to Linear.

Workspace/team: [x]
Linear issue: [x]
GitHub repo/PR: [x]
Expected: PR should link / move issue to In Progress / close issue on merge
Actual: [x]
Checks completed:
- Integration enabled: yes/no/unknown
- Repo connected: yes/no/unknown
- Issue ID present in branch/title/body: yes/no
- Keyword used: Part of / Closes / none
- PR state: draft/open/merged
Impact: [blocking workflow? one user? multiple?]
```

---

### Case B — “My issue is stuck in triage.”

**Likely meaning:** The issue has been created but not accepted into the team’s active workflow.

**Check:**

1. Is Triage enabled?
2. Which team owns the triage queue?
3. Was the issue accepted?
4. Is there a goalie/triage owner?
5. Is the user expecting backlog behavior instead of triage intake?

**Reply pattern:**

> It looks like this issue is still in Triage, which means it has been received but not yet accepted into the team’s active workflow. Triage is intentional intake, so the next step is for the team’s triage owner to review, prioritize, and accept or reroute it.

---

### Case C — “My teammate cannot see the issue.”

**Check:**

1. Workspace membership.
2. Team membership.
3. Private team/project permissions.
4. Guest access limitations.
5. View filters.
6. Archived/completed issue visibility.

**Reply pattern:**

> This is usually a permission or view-scope issue rather than the issue being missing. I would first confirm whether your teammate has access to the workspace and the specific team/project, then check whether their current view has filters hiding completed, archived, or unassigned issues.

---

### Case D — “Status did not update after opening a PR.”

**Check:**

1. Was PR linked to the issue?
2. Are status automations configured?
3. Is PR in draft?
4. Was reviewer added?
5. Did the branch/PR include issue ID?
6. Sync delay?

**Reply pattern:**

> A linked PR does not always mean every status transition will happen automatically; it depends on the workspace’s GitHub integration and workflow settings. I would first verify that the PR is linked to the issue, then check whether the automation is configured to move linked issues when PRs open, enter review, return to draft, or merge.

---

### Case E — “Where is the preview deploy?”

**Check:**

1. Vercel/Netlify integration connected?
2. Preview attached to PR?
3. PR linked to Linear issue?
4. User has permission to view preview?
5. Link added only in GitHub comment instead of surfaced through integration?

**Reply pattern:**

> Preview links usually surface on the Linear issue when the deployment provider is connected and the PR is linked to the issue. If the preview exists in GitHub but not Linear, I would check whether the PR is properly linked and whether the deployment integration has permission to attach previews to the issue.

## 11. Support writing templates

### Template 1 — First response

```md
Thanks for the context — I can help look into this.

From what you described, it sounds like [plain-language summary]. The first things I’d check are:

1. [check one]
2. [check two]
3. [check three]

Could you send [specific artifact: issue ID, PR link, screenshot, workspace/team name]? That should let us narrow down whether this is expected behavior, configuration, or a sync issue.
```

### Template 2 — Reproduction request

```md
I’d like to reproduce this as closely as possible.

Could you share:

- The Linear issue ID or URL
- The GitHub PR/repo link, if relevant
- What you expected to happen
- What actually happened
- Whether this happens for one issue or multiple issues

Once I have that, I can separate whether this is a workflow/configuration issue or something we should escalate.
```

### Template 3 — Expected behavior explanation

```md
This appears to be expected behavior based on how [feature] works today.

The key distinction is:

- [Option A] does [x]
- [Option B] does [y]

In your case, [specific explanation]. The best next step is [action]. I agree this can be easy to miss, so I’ll also note this as product/documentation feedback.
```

## 12. Work trial operating principles

If you reach the work trial, act like a teammate, not a student waiting for instructions.

### Principles

1. **Start by restating the problem.**
2. **Make assumptions visible.**
3. **Use small, clear deliverables.**
4. **Communicate progress before being asked.**
5. **Ask precise questions, not broad ones.**
6. **Show judgment, not just effort.**
7. **Write like support: calm, direct, human, useful.**
8. **Leave behind reusable documentation.**
9. **Escalate with evidence.**
10. **Respect time: concise updates, clean artifacts.**

### Work trial daily rhythm

#### Day 0 / Start

- Read the prompt twice.
- Identify expected output.
- Write a short plan.
- Ask only blocking questions.

#### Day 1

- Produce first useful draft.
- Do not wait for perfection.
- Share assumptions and risks.

#### Day 2

- Refine based on feedback.
- Add examples, edge cases, and support tone.
- Show what changed.

#### Day 3–5, if applicable

- Polish.
- Add final summary.
- Include “what I would do next if I had more time.”

### Work trial update template

```md
Quick update:

What I completed:
- [x]
- [y]

What I am checking next:
- [z]

Current assumption:
- [assumption]

Potential risk / open question:
- [question]
```

## 14. What to avoid

### Avoid overexplaining

Bad pattern:

> “This connects to my whole philosophy of AI and work and building and support and maybe GitHub and maybe...”

Better:

> “The issue is likely one of three things: integration, reference format, or permissions. I would check those in that order.”

### Avoid sounding like you need too much structure

Bad:

> “I need someone to tell me exactly what to do.”

Better:

> “I like clear goals, then I can create my own operating structure around the work.”

### Avoid apologizing for your background

Bad:

> “I know I don’t have the perfect resume.”

Better:

> “My background is nontraditional, but the through-line is customer judgment, operational calm, and fast technical learning.”

### Avoid claiming mastery you do not have

Bad:

> “I am an expert in Linear integrations.”

Better:

> “I have been actively practicing the workflows: issues, triage, GitHub linking, PR sync, and support-style troubleshooting.”
